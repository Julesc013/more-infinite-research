# 3.0.0 Transition Checkpoint 2

Updated: 2026-07-07

This checkpoint records the second named MIR 3 transition state after the shell/governance baseline. It does not declare `3.0.0` release readiness. It names the current safe midpoint so later compiler migration can be compared against a stable, described state.

## Completed In This Checkpoint

- `.mir/` governance manifests exist and are populated enough to lint docs, module boundaries, stream records, compatibility claims, fixtures, branch policy, and agent routing.
- Root Factorio entrypoints route through `prototypes/mir/stage/`.
- `control.lua` is retained only for real runtime scripted-tech behavior and is isolated from prototype generation.
- `StreamSpec` and `DecisionRecord` modules exist as MIR-domain records.
- Generated technology emission now passes through `prototypes/mir/emit/technology_builder.lua`.
- Air Scrubbing has a declarative compatibility overlay under `prototypes/mir/compatibility/overlays/`.
- `.mir/streams.yml` records Air Scrubbing and ATAN Ash generated stream identities.
- `.mir/compatibility.yml` records public compatibility targets and claim levels for the current proof set.
- The local mod-library catalog gate exists and passes against the Factorio `2.1` local library used for this checkpoint.

## Known Remaining Gaps

- `prototypes/mir/legacy/data_final_fixes.lua` still owns the main data-final-fixes orchestration.
- Air Scrubbing is declared as MIR 3 policy data, but the legacy Air Scrubbing emitter still owns its behavior path.
- `StreamSpec` and `DecisionRecord` are real but still thin transition records.
- `platform/`, `index/`, `graph/`, `classify/`, `policy/`, and `report/` are not yet fully materialized as compiler layers.
- `claim_registry.lua` is still a stub and must either mirror the runtime/report claim subset or explicitly declare `.mir/compatibility.yml` as the only claim authority.

## Next Work

The next transition work should reduce legacy ownership without broad behavior changes:

1. Add a Factorio `data.raw` platform adapter and route emission through it.
2. Make the compatibility claim registry explicit.
3. Route Air Scrubbing generation through overlay policy, capability planning, `StreamSpec`, and the emitter.
4. Add a report layer that can export `DecisionRecord` data while preserving existing diagnostics.
5. Continue using regression row/hash comparison after each behavior-adjacent compiler slice.
