# Adding A Compatibility Pack

Run the generic compiler first and save complete coverage accounting for one independently pinned ecosystem. A pack is justified only by measured ambiguity that cannot safely become a generic structural rule.

Create one schema-2 data packet with exact applicability and target lines. Keep aliases, includes, excludes, family hints, science roles, owner claims, family authorizations, candidate seeds, and risk overrides as narrow as the evidence permits. An allow-reviewed risk override requires a permitted soft signal, exact recipe selector, exact applicable mod version, declared evidence IDs, and a dedicated positive fixture. It can never override effective productivity denial, recycling, probability, catalyst, self-return, parameter, cap, or owner facts.

Use `family_authorizations` only to approve creation when the player's `Require reviewed data for new research` control is enabled. Each row names an existing family or stable stream, `action = generate`, fixture evidence, and a claim boundary. Use `candidate_seeds` when an exact reviewed recipe is outside structural discovery; name the recipe, existing FamilyRule, stable stream, optional exact item and tier, positive change, and evidence. Seeds add candidates, never technology identities.

Add focused fixtures for every refinement and run an exact candidate-archive campaign with root versions, dependency closure, archive hashes, source commit, lock fingerprint, actual executed roots, result, and claim level. Begin at `loads`; only raise the claim after coverage, progression, upgrade, and interactive evidence supports it.

Test-only packets belong in fixtures through the internal `mod-data` transport. Do not add fixture identities to production profiles. Update `.mir/compatibility.yml`, the relevant compatibility page, `.mir/docs.yml`, and the campaign evidence record. Commit one ecosystem pack at a time.
