# 2.2.0 Feature Intake From The July 2026 Suggestion Transcript

Source context:

- The 2026-07-05 idea-mod audit remains the authority for role classification.
- The July 2026 suggestions transcript is useful as a demand and risk review.
- This note records what MIR accepts from that transcript and what it rejects.

This is a planning note for the `2.2.0` development line. It does not change the
`2.1.5` release target. Finish the `2.1.5` docs polish, validation, sync to
`main`, and release first.

## Product Rule

There is one mod:

```text
More Infinite Research
```

Do not introduce separate product names, companion product names, "Extended"
branding, or settings labels that imply the user has to understand a product
family. The transcript's split between core, extended, runtime experiments, and
companions should be translated into normal MIR language:

| Transcript language | MIR language |
| --- | --- |
| Core | Default behavior. |
| Extended | Opt-in startup setting, if the feature fits MIR and has tests. |
| Runtime experiment | Deferred runtime feature, usually rejected until performance and migration proof exist. |
| Companion or separate mod | Not planned for MIR unless a future project decision explicitly changes scope. |

The settings page should stay easy to scan:

- Prefer one clear setting per feature family, not many per-mod toggles.
- Use direct labels such as "Recipe productivity cap behavior" or "Native research overlap behavior".
- Avoid vague labels such as "Extended", "experimental pack", "danger mode", "compatibility suite", or named bundles.
- Use descriptions to say what changes balance, what requires restart, and what remains off by default.
- Do not expose a setting until the implementation and validation fixtures exist.

## Existing MIR Coverage

Many transcript ideas are already covered by current MIR behavior in a better or
narrower form:

| Area | Existing MIR form |
| --- | --- |
| Infinite productivity streams | MIR already generates explicit fixture-backed recipe productivity streams instead of broad name-based mutation. |
| Per-technology control | Each generated stream already has enable, cost, growth, max-level, and research-time startup settings. |
| Exact duplicate cleanup | `2.1.5` has guarded known-competitor cleanup for exact infinite recipe-productivity owners. |
| Native duplicate avoidance | `2.1.5` has effect-proven skips for lab productivity and worker robot battery duplicates. |
| Cap awareness | `2.1.5` emits recipe-cap diagnostics and compatibility observation artifacts without changing caps. |
| Planner foundation | `2.1.5` emits diagnostics-only compatibility role, planner, and recipe-cap rows. |
| Compatibility reporting | Audit conversion now writes grouped failures, profile candidates, missing dependencies, and compatibility observations. |
| Startup-only prototype settings | The pipeline extent multiplier is the existing pattern: default unchanged, no prototype scan when off. |

This means `2.2.0` should not start by adding a large settings surface. It should
first turn the diagnostics surfaces into a stronger planner and then add only the
small number of behavior changes that are fixture-backed.

## 2.2.0 Working Policy

Use this rule for every transcript feature:

```text
Default MIR behavior stays conservative.
Opt-in behavior is allowed only when it is deterministic, startup-gated when it
mutates prototypes, fixture-backed, documented, and silent when disabled.
```

"Disabled" must mean:

- no prototype mutation;
- no broad prototype scan unless a diagnostics setting explicitly needs it;
- no runtime event handlers;
- no runtime storage;
- no public replacement claim;
- no setting-page clutter for unimplemented ideas.

Prototype changes belong in startup settings. Runtime settings are only for real
runtime logic. Do not make a live toggle for:

- recipe productivity eligibility;
- beacon allowed effects;
- machine base productivity;
- recipe `maximum_productivity`;
- technology cost formulas;
- lab prototype changes.

## Feature Family Decisions

| Family | 2.2.0 decision | Notes |
| --- | --- | --- |
| Compatibility planner | Build next. | Promote the `2.1.5` diagnostics-only rows into a structured planner/registry with roles, actions, non-actions, warning classes, and claim boundaries. |
| Cap diagnostics | Continue. | `2.1.5` warning rows are the base. Add useful-level estimates and better conflict reporting before any cap mutation setting. |
| Cap behavior | Design first, maybe implement one narrow option. | Candidate setting: "Recipe productivity cap behavior". Default remains vanilla. Do not silently remove caps or convert infinite techs to finite. |
| Productivity rule changes | Do not ship broad mutation first. | High demand, high risk. Start with external rule-mutator detection and loop-risk diagnostics. A later opt-in setting needs allowlists and a denylist. |
| Loop-risk scanner | Good 2.2.0 diagnostic slice. | Needed before any option that expands productivity eligibility or raises/removes caps. |
| Machine base productivity | Lower priority. | If added, use explicit machine allowlists and warnings when another mod already changed the same machine. |
| Runtime production-stat productivity | Not for 2.2.0. | Requires save-state, migration, performance, reset commands, and uninstall policy. No idle event handlers. |
| Productivity-through-science | Not for 2.2.0. | Runtime force/recipe behavior does not fit the next release unless a separate design proves performance and save behavior. |
| Solar or accumulator entity replacement | Reject from MIR 2.2.0. | Popularity is not fit. Do not turn MIR into an energy entity replacement mod. |
| Research cost controls | Coexist and diagnose only. | Useful adjacent tools, but not productivity ownership. Do not add broad research-cost mutation in 2.2.0. |
| Native direct modifiers | Small policy only. | Lab productivity/speed, worker robot battery/speed, and mining productivity are plausible. Avoid combat, character, reach, inventory, and broad bundled upgrades. |
| Ore crushing productivity | Good 2.2.0 stream candidate. | Requires Crushing Industry recipe-ID fixtures and no forced balance assumptions. |
| Air Scrubbing filters | Good 2.2.0 stream candidate. | Only clean pollution/spore filter recipes. Do not touch scrubbing or cleaning recipes. |
| Tile and surface productivity | Policy before implementation. | Asphalt, concrete, landfill, and foundation values differ widely in source mods. Use conservative defaults and exact cleanup only. |
| Overhaul material families | One narrow family at most. | Pick a concrete recipe family with visible IDs. Do not build a generic productivity generator. |
| Content, radar, lab, mining, logistics mods | Compatibility only. | Add fixtures and lab/science diagnostics where they intersect MIR. Do not recreate entities, labs, radars, belts, drills, pumps, or surveillance systems. |

## Candidate Setting Shape

These are planning names, not implementation commitments. Final labels should be
checked against locale and settings ordering before any code ships.

| Setting idea | Default | Allowed shape |
| --- | --- | --- |
| Recipe productivity cap behavior | Vanilla caps | `vanilla`, `warn`, `limit MIR levels`, maybe one explicit cap-raise option after proof. |
| Recipe productivity rule changes | Off | Start with `off` and maybe `selected recipes` only. Do not start with broad/everything behavior. |
| Native research overlap behavior | Auto safe | `auto safe`, `prefer MIR`, `prefer other mods`, `warn only`, if the planner proves each mode. |
| Rule mutation report | Off or tied to diagnostics | Log external rule mutators and loop-risk warnings without changing prototypes. |

Settings should not be named after source mods. Source-mod names belong in audit
reports and compatibility docs, not the player settings page.

## 2.2.0 Start Order

After `2.1.5` is released:

1. Open the `2.2.0` development line and update version/changelog placeholders.
2. Turn the diagnostics-only planner into a structured planner object used by reporting and future mutation gates.
3. Add external rule-mutator detection as diagnostics, especially recipe productivity eligibility, beacon effects, machine base productivity, cap changes, and research-cost changes.
4. Add loop-risk diagnostics for recycling, catalyst returns, cleaning/recovery, scrubbing, spoilage, self-product recipes, and obvious voiding loops.
5. Improve cap diagnostics with useful-level estimates and conflict notes.
6. Build the Crushing Industry ore-crushing stream only after recipe-ID fixtures pass.
7. Build the Air Scrubbing clean-filter stream only after fixtures prove scrubbing and cleaning recipes are excluded.
8. Decide the first native modifier overlap policy, kept to a tiny set of native effects.
9. Revisit tile/surface productivity only after a value/default policy is written.
10. Keep runtime productivity, solar/entity replacement, research-cost mutation, and large content recreation out of the `2.2.0` implementation queue.

## 2026-07-06 Implementation Status

The first `2.2.0` development slice is now implemented as a report-only compiler
spine:

- `prototypes/lib/facts/registry.lua` indexes typed recipe, technology, machine,
  lab, owner, and rule-surface facts from the loaded prototype world.
- `prototypes/planner/compiler.lua` emits fact-registry summaries, owner
  summaries, lab-matrix rows, compiler decision rows, loop-risk rows, and
  rule-surface rows when generation diagnostics are enabled.
- Recipe-cap diagnostics now include useful-level estimates for warning rows.
- Compatibility observation conversion preserves the new compiler rows in
  `compat-observations.*` and the overnight summary shows their counts.
- The new path does not add broad automatic stream generation, cap mutation,
  beacon mutation, runtime productivity, research-cost mutation, or new product
  branding.

Remaining `2.2.0` behavior work should start from fixtures and exact recipe IDs:
Crushing Industry ore crushing, Air Scrubbing clean filters, one narrow material
family, and a small native modifier policy only after the report rows show safe
targets.
