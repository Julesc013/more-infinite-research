# MIR 3.0.5 Settings Inventory

On the Factorio 2.1 target, MIR 3.0.5 registers 477 startup settings: 11
dropdowns, 85 checkboxes, 380 numeric fields, and one profile-import text
field. All are startup settings, so changing one requires returning to the main
menu or restarting Factorio before prototypes are rebuilt.

Every dropdown has an explicit neutral path. A neutral selection leaves the
engine or MIR's configured baseline unchanged and bypasses the corresponding
optional transformation. Existing enum IDs remain accepted so saved settings,
portable profiles, and supported backports do not lose intent.

The dropdown is only the convenient UI. A portable profile may provide a JSON
number for any numeric dropdown, using the percentage displayed to the player.
For example, `123.45` means 123.45%, and `-83.25` means -83.25%. This supports
valid decimal values, values between menu steps, and advanced values outside
the curated ladder without weakening Factorio's registered dropdown.

## Dropdowns

| Setting | Default | Neutral/bypass | Options and behavior |
| --- | --- | --- | --- |
| `mir-science-pack-ingredient-policy` | Configured packs | `configured` | Configured packs, add space, add space + promethium, Space Age progression, official progression, modded progression, all official, or all lab packs. Configured uses the stream ingredients without optional expansion. |
| `mir-lab-incompatibility-policy` | Reduce | `engine-default` | Reduce to a researchable subset, skip unsupported technologies, or no MIR adjustment. The neutral path never rewrites ingredients: it preserves an exactly researchable set or safely omits the generated technology. |
| `mir-adjust-vanilla-weapon-speed-techs` | Conditional | `off` | Off, only with exact replacement coverage, or always remove MIR's overlapping rocket/cannon effects. Off disables MIR's optional cleanup; finite vanilla technologies are never changed. |
| `mir-pipeline-extent-multiplier` | 100% | `100` | 25%, 50%, 75%, 100%, 125%, 150%, 200%, 250%, 300%, 400%, 500%, 750%, or 1000%. At 100%, MIR does not load the pipeline mutation pass. |
| `mir-prototype-productivity-cap` | Engine +300% | `engine-default` | +25%, +50%, +75%, +100%, +125%, +150%, +200%, +250%, engine +300%, +400%, +500%, +750%, +1000%, +2500%, +5000%, +10000%, +25000%, or +100000%. |
| `mir-recycling-return-chance` | Engine unchanged | `engine-default` | Unchanged, match productivity cap, or fixed 20%, 15%, 12.5%, 10%, 7.5%, 5%, 2.5%, 1%, 0.5%, or 0.1%. The duplicate fixed 25% row is hidden, but legacy `percent-25` and numeric `25` profile imports remain valid. Only hidden generated recycling recipes are eligible. |
| `mir-prototype-efficiency-cap` | Engine -80% | `engine-default` | -25%, -50%, -75%, engine -80%, -90%, -95%, -99%, -99.9%, or -99.99% energy-use lower limit. |
| `mir-prototype-pollution-cap` | Engine -80% | `engine-default` | -25%, -50%, -75%, engine -80%, -90%, -95%, -99%, -99.9%, or -99.99% pollution lower limit. |
| `mir-prototype-speed-floor` | Engine -80% | `engine-default` | -25%, -50%, -75%, engine -80%, -90%, -95%, -99%, -99.9%, or -99.99% negative speed floor. This is independent from the positive speed cap. |
| `mir-prototype-speed-cap` | Engine +100000% | `engine-default` | Engine +100000% unchanged, or explicit +25%, +50%, +75%, +100%, +125%, +150%, +200%, +250%, +300%, +400%, +500%, +750%, +1000%, +2500%, +5000%, +10000%, or +25000%. |
| `mir-prototype-quality-cap` | Engine +100000% | `engine-default` | Engine +100000% unchanged, or explicit +25%, +50%, +75%, +100%, +125%, +150%, +200%, +250%, +300%, +400%, +500%, +750%, +1000%, +2500%, +5000%, +10000%, or +25000%. |

The automatic recycling choice uses `min(25%, 1 / (1 + productivity bonus))`.
For example, +400% productivity means five total outputs and a 20% return;
+1000% means eleven total outputs and about 9.09%. It never increases the
normal 25% recycler return.

## Global Checkboxes

| Setting | Default | Effect when enabled |
| --- | --- | --- |
| `ips-require-space-gate` | Off | Requires the applicable game-completion gate before generated research. |
| `mir-prefer-this-mod-for-competing-techs` | On | Lets MIR replace known competing research only after complete semantic coverage is proved. |
| `mir-use-installed-space-age-icons` | Off | Uses installed Space Age icons where the active target permits them. |
| `mir-productivity-cap-self-recycling-only` | Off | Derives the unrestricted cap from `1 / recycler return - 1`; only proved non-generative self-recycling recipes may exceed it. |
| `mir-unrestricted-modules` | Off | Opens discovered module categories/effects on existing recipe and receiver slots; it does not add slots. |
| `mir-prototype-positive-power-floor` | Off | Changes explicit active-use `0W` prototype values to `1W`. |
| `mir-debug-generation-report` | Off | Logs generated and skipped technology decisions. |
| `mir-debug-recipe-matches` | Off | Logs recipe-matching diagnostics. |
| `mir-debug-scripted-effects` | Off | Logs bounded scripted-effect diagnostics. |

## Generated Technology Controls

MIR has 70 generated stream groups. Each group contributes one enable checkbox
and five numeric fields: base cost, growth factor, maximum level, research time,
and effect per level. That is 70 checkboxes and 350 numeric fields.

MIR also has six base-continuation groups: braking force, inserter capacity,
laser shooting speed, lab research speed, weapon shooting speed, and worker
robot cargo size. Each contributes the same one checkbox plus five numeric
fields, for six checkboxes and 30 numeric fields.

Together with the nine global checkboxes, these groups produce the 85-checkbox
and 380-numeric-field totals. Maximum level `0` means infinite. Effect-per-level
values scale MIR-emitted effects from their canonical primary-tier anchor and
do not rewrite external technology owners.

## Profile Import

`mir-settings-profile-import` is the one text field. Paste a `MIRSET1:` profile
and restart to use it as an effective startup-setting override. Invalid or
unsupported entries are ignored rather than coerced, and the import field is
never nested inside exported profiles.

Custom numeric dropdown import bounds are:

| Settings | Imported percentage range |
| --- | --- |
| Pipeline extent | 0.1% to 100000% |
| Productivity cap | 0% to 100000% |
| Recycler return | 0% to 25% |
| Efficiency, pollution, and speed floors | -99.99% to 0% |
| Speed and quality caps | 0% to 100000% |

For example, raw profile JSON may contain:

```json
{
  "schema": 1,
  "kind": "mir-settings-profile",
  "settings": {
    "mir-pipeline-extent-multiplier": 123.45,
    "mir-prototype-productivity-cap": 4321.5,
    "mir-recycling-return-chance": 12.34,
    "mir-prototype-speed-floor": -83.25
  }
}
```

Raw JSON is accepted for advanced profile authoring; normal exports remain the
canonical `MIRSET1:` encoded form. Non-finite, wrong-sign, and out-of-range
values are rejected and the normal dropdown value remains effective.
