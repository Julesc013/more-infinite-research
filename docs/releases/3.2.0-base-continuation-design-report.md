# 3.2.0 Base Continuation Design Report

## Outcome

Base-chain continuations now use the common schema-2 `TechnologyDesign` authority. The allowlisted continuation builder still derives the initial safe alternative from finalized vanilla technology facts, but CompilationPlan rebuilds the accepted design after effect sanitation and overlap policy. Emission consumes only the final design projection through the same adapter used by fixed and automatic streams.

Six released chain identities are now explicit in `generated_continuation_manifest.json`: braking force, inserter capacity, laser shooting speed, research speed, weapon shooting speed, and worker robot storage. Each row binds a stable manifest ID, chain key, next-level identity pattern, released state, and stable migration policy. This does not make the numeric next level independent of the finalized upstream chain; it makes the authority for deriving that level explicit and reviewable.

TechnologyDesign now includes materialization kind and preserves either single-icon or layered-icon presentation fields. `create`, `continuation`, `patch-existing`, and `diagnose` are governed values. The shared prototype adapter accepts only create and continuation designs; patch-existing remains transaction-owned.

## Validation and caught regressions

Architecture, policy, schema-drift, compiler-contract, and exact base-extension-boundary gates passed. The exact boundary scenario preserved a fixture-provided finite `research-speed-7` and emitted the planned next continuation without replacement.

The approved-delta producer rejected two intermediate attempts. The first leaked design metadata into the public generated registry, producing 847 unapproved rows. The second synthesized an order value for continuations whose base prototype had no order, producing 35 unapproved presentation changes. Both defects were removed before acceptance.

The final deterministic Optimal archive SHA-256 is `7BA3D90307A7C62FB57B6E2D2791C9727E4A6CC4591CED31D70A6EB55099697B`. Fourteen fresh Factorio 2.1.10 comparison loads produced 25 intentional differences, zero unapproved differences, and zero technology-field differences. The one new classified row is the settings-source fingerprint for stable manifest IDs; setting values, registry contracts, runtime namespaces, and migrations did not change.

## Boundary

This is a continuation-only IR migration. Native-owner adoption is represented by a design record but its transaction does not yet consume that design as the patch-existing authority. Candidate, qualification, approval, promotion, and migration records remain follow-on work.
