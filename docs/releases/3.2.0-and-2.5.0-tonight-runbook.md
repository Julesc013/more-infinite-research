# MIR 3.2.0 and 2.5.0 Release-Night Runbook

This runbook separates the decisions that can be made after the C15 playtest:

1. keep the exact C15 package and release it after its remaining gates pass;
2. make a package-visible correction, create a later 3.2.0 candidate, and requalify that package; or
3. defer publication and continue tuning without weakening any gate.

MIR 2.5.0 is a subsequent Factorio 2.0 backport from the frozen 3.2.0 source. It is not the same ZIP, cannot inherit 3.2.0 runtime evidence, and is not released directly from `dev` or `main`.

## Work that can finish unattended

Before interactive playtesting:

- run the exact approved-delta exporter and require zero unapproved changes;
- run the complete static suite;
- run the paired, balanced performance campaign against MIR 3.1.9;
- materialize a fresh `full` verification plan with evidence reuse disabled;
- execute every machine-verifiable row in that exact plan;
- preserve compact evidence and delete disposable per-run directories;
- keep all `testmods_*` libraries;
- push qualification-only fixes and evidence to `dev`.

An automated pass does not authorize the package for publication. Manual package review, protected qualification, seal creation, promotion, tagging, and publication remain explicit release-manager actions.

## Interactive C15 review

Use `docs/releases/3.2.0-c15-playtest-guide.md` as the detailed in-game order. The minimum release-night sample is:

1. base-only new game and a copied 3.1.9 save;
2. Space Age with native steel ownership;
3. K2SO plus the enhancement fork, including non-red science on advanced circuits;
4. the governed large Space Age ecosystem;
5. the smallest isolation ring for every suspicious observation.

There is one new stable MIR-generated technology identity in C15: `recipe-prod-research_steel-1` in environments without a suitable native steel owner. Space Age must retain `steel-plate-productivity` as the single owner. Automatic assembler, lab, chemistry, biological, recovery, matter, catalyst, and other diagnostic candidate families remain experimental and must not be described as newly shipped stable technologies.

For every technology considered for promotion, review:

- stable identity and family manifest;
- exact recipe or direct-effect membership;
- owner uniqueness;
- prerequisites and reachability;
- science packs and an available accepting lab;
- first-level cost, time, growth, and per-level effect;
- icon, locale, tree position, arrows, and settings exposure;
- new-save, reload, upgrade, partial-research, and migration behavior;
- applicability and quality-assessment evidence across its claimed mod rings.

Do not batch-promote every diagnostic candidate. Promotion is an individual public-contract decision. It requires the lifecycle admission gate and a package-visible rebuild, so any promotion after C15 creates a later candidate.

## Decision after playtesting

### Release C15

Keep C15 only when all observations are acceptable and the exact archive still matches the frozen identity. Complete the following in order:

1. capture the package-focused interactive review packet;
2. complete and sign the schema-2 manual attestation for the exact ZIP and exact Factorio binary;
3. commit the attestation as a qualification-only revision;
4. materialize the final no-reuse full plan at that qualification commit;
5. dispatch the protected Factorio 2.1 full-qualification workflow;
6. require every required capsule and the aggregate gate to pass;
7. create the schema-4 candidate seal and independently run `check-seal`;
8. run the promotion check against the same archive bytes;
9. promote `dev` to `main` without changing the package;
10. tag the final release-evidence commit;
11. publish the already-sealed ZIP without rebuilding it;
12. download both published artifacts and verify archive SHA-256 and size.

The public release is not ready if the only missing row is manual review. A deliberately absent human attestation is a correct blocked gate, not an automated failure to bypass.

### Create a later 3.2.0 candidate

Create a later candidate when a demonstrated package-level blocker requires a change to Lua, locale, settings, migration, package metadata, assets, packaged documentation, stable streams, or package-building behavior that changes ZIP bytes.

After the change:

1. record why C15 was superseded;
2. rebuild deterministically and record the new candidate identity;
3. rerun the narrow reproducer first;
4. regenerate all package-bound authority and evidence;
5. repeat the complete automated, manual, protected, and seal gates.

Do not preserve the C15 name or hashes cosmetically.

### Continue tuning without release

It is valid to keep the package frozen while collecting playtest bundles and deciding whether subjective balance observations justify a new candidate. Never loosen performance, sanitation, reachability, or evidence-trust policy to make an unresolved observation disappear.

All boxes must refer to the same candidate bytes:

- [ ] deterministic package identity matches the release authority;
- [ ] package composition contains no forbidden paths;
- [ ] exact approved delta passes;
- [ ] complete static suite passes;
- [ ] base and Space Age exact-archive loads pass;
- [ ] full runtime scenario matrix passes;
- [ ] five-row MIR 3.1.9 upgrade matrix passes;
- [ ] ecosystem and sanitation campaigns pass their exact budgets;
- [ ] paired performance campaign passes every lane and counter bound;
- [ ] evidence portability and aggregate verification gate pass;
- [ ] release notes and compatibility claims match the actual package;
- [ ] exact package manual attestation passes;
- [ ] protected full no-reuse qualification passes;

      sealed ZIP;
- [ ] downloaded public ZIP matches the sealed hash and byte count.

## MIR 2.5.0 backport order

Start the 2.5.0 backport only after the canonical 3.2.0 source and public behavior are frozen. Use a separate worktree for `tmp/2.0`; do not disturb the qualified 2.1 checkout.

1. update the `tmp/2.0` backport source lock to the exact frozen canonical 3.2.0 anchor;
2. reconstruct the target from that lock and record every deliberate Factorio 2.0 cut or adapter;
3. set MIR version `2.5.0` and Factorio line `2.0` only in the target branch;
4. exclude or guard 2.1-only prototype fields, effects, official-mod assumptions, and metadata;
5. preserve stable stream IDs wherever the public concept exists on 2.0;
6. update target fixtures, target profiles, approved delta, release notes, and compatibility claims;
7. build the exact 2.5.0 ZIP from `tmp/2.0`;
8. materialize the Factorio 2.0 verification plan using the exact 2.0 binary, exact 2.4.9 prior ZIP, and `testmods_2.0`;
9. run capability, settings, deterministic-package, static, exact runtime, upgrade/retention, ecosystem, performance, and evidence gates;
10. manually review the exact 2.5.0 package;
11. run protected target qualification, seal, and independent seal check;
12. promote the validated target branch to `legacy`;
13. tag and publish the already-sealed 2.5.0 ZIP;
14. download it and verify the public archive identity.

The target helper is:

```powershell
.\scripts\Invoke-MIRBackportQualification.ps1 `
  -Action qualify `
  -FactorioBin '<exact Factorio 2.0 binary>' `
  -CandidateZip .\dist\more-infinite-research_2.5.0.zip `
  -PriorZip .\dist\more-infinite-research_2.4.9.zip
```

Run `-Action seal` only from a clean, fully qualified target checkout, then run `-Action check-seal` independently. The backport seal is target-specific; the 3.2.0 seal cannot be reused.

## 2.5.0 release blockers

Do not call 2.5.0 release-ready until:

- the frozen 3.2.0 canonical anchor is recorded;
- the target reconstruction and backport-source lock are current;
- Factorio 2.0 metadata and API cuts are reviewed;
- the exact 2.5.0 ZIP has its own approved delta and runtime evidence;
- upgrade from exact 2.4.9 preserves settings, research, and stable identities;
- public claims are limited to tested 2.0 fixtures and named load checks;
- manual and protected target qualification pass;
- `legacy` contains the exact sealed target source and package.

If 3.2.0 changes after 2.5.0 work starts, classify the difference explicitly. Update the backport lock and repeat affected target evidence when it changes the intended 2.5.0 behavior. Do not silently inherit a moving `dev` head.

## Storage discipline

Keep:

- exact current and prior release ZIPs;
- compact governed evidence and candidate descriptors;
- final plans, summaries, seals, and manual attestations;
- `testmods_*` libraries and explicitly governed fixture caches.

Remove after the producing command finishes and its useful result is retained:

- extracted candidate copies;
- temporary Factorio user-data directories;
- duplicate mod-library staging copies;
- obsolete per-invocation performance and scenario roots;
- stale build/package scratch trees;
- superseded untracked logs and abandoned plans.

Run the storage audit after long campaigns. Never delete a `testmods_*` directory as automated cleanup, and never remove evidence referenced by a current release authority or seal.
