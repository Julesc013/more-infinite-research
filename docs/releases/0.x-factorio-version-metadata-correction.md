# MIR 0.x Factorio Version Metadata Correction

The original MIR 0.12.0 through 0.6.0 museum archives contained a valid MIR `version` but omitted the explicit `factorio_version` field. The upload validator rejected each archive with `Invalid info.json: {'factorio_version': ['Missing data for required field.']}`.

The defect was in the shared museum compiler, not in seven independently authored files. The compiler now emits an exact target-matching `factorio_version`, and its rendered-source validator rejects both a missing field and a mismatched field. The fix therefore applies deterministically to every museum target without allowing one-off package edits.

## Validation

- All seven targets built deterministically with four permitted release entries and no development-only paths.
- The compiler's 26 negative and invariant cases passed.
- Every exact corrected archive passed the recorded matching-binary load and reload route: ZIP-native for 0.12 through 0.9 and extracted-directory for 0.8 through 0.6.
- Every target qualification record and immutable candidate seal was regenerated and passed.
- GitHub reported the same SHA-256 as the local sealed ZIP for every uploaded asset.
- The staged `dev` tree passed integrity for all 16 published source snapshots and all 46 root archives.
- The complete `Invoke-MIRValidation.ps1 -StaticOnly` gate passed; its two development-package builds were byte-identical with SHA-256 `3DBCCE7903FB6182D34C83E67E2DDF277E0BB8B8FD8042C2C238A1F78BB1129D`.
- Manual technology-tree, icon, locale-fit, save-UI, and balance review remains `PENDING-MAINTAINER`.
- Mod Portal upload and acceptance remain maintainer actions; RC readiness does not claim that an external service accepts these historical Factorio lines.

## Durable Lessons

- Required compatibility metadata belongs in the shared generator contract and its negative tests.
- A package validator must check semantic target equality, not merely the presence of a JSON key.
- Runtime proof, deterministic packaging, public asset verification, and external portal eligibility are separate claims.
- Post-publication branch ledgers may follow a sealed tag because release and asset IDs do not exist until publication; the tag remains pinned to the sealed candidate.
