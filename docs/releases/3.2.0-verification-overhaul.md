# MIR 3.2.0 Persistent Verification Overhaul

MIR 3.2.0 replaces commit-shaped test repetition with persistent, content-addressed verification evidence. The release preserves public setting IDs, generated technology IDs, migrations, and runtime namespaces. Its package also contains the portable target-capability guard already integrated on `dev`, so the 3.2.0 data-domain hash differs from the released 3.1.9 archive and fresh gameplay qualification is required even though Factorio 2.1 continues to support and emit the same mod-data surfaces.

## Contract

`validation/tests.yml` owns stable test IDs and scenario-matrix templates. `validation/domains.yml` owns package-domain classification and scenario dependencies. `validation/profiles/factorio-2.1.json` and `validation/profiles/factorio-2.0.json` bind target-specific policy, deterministic seed, evidence lifetime, and upgrade identity. `scripts/Invoke-MIRAssurance.ps1` remains the single authoritative verifier, with `tools/mir_verify/Invoke-MIRVerify.ps1` as a forwarding entrypoint.

Every canonical schema-4 plan records one exact fingerprint per required test instance and assigns `REUSE`, `WAIT`, `RUN`, or `INVALID`. Plan material binds the complete profile-derived required set, current test definitions, source commit and tree, exact candidate descriptor, policy authorities, and effective inputs. Workers and the aggregate gate reconstruct that material and reject empty, truncated, altered, duplicate, or stale plans. `REUSE` requires a same-trust-class schema-4 passing capsule with the exact test ID, target, definition, effective inputs, structured result, artifact hashes, producer policy, and result digest. `WAIT` means another live worker owns the same fingerprint; the worker is adopted rather than cancelled. `INVALID` means stored material exists but is failed, blocked, malformed, untrusted, stale, or digest-mismatched. Unknown inputs invalidate conservatively.

## Factorio Layers

| Layer | Scope | Artifact rule |
| --- | --- | --- |
| `F0` | Static contracts, architecture, compiler, settings, locale, balance | Repository-domain fingerprints |
| `F1` | Deterministic package construction | Package-source fingerprint |
| `F2` | Exact packaged ZIP loading | Exact archive hash |
| `F3` | Gameplay and data-stage scenarios | Declared package domains, not the whole ZIP |
| `F4` | Configuration changes, upgrade, ecosystem, approved delta, performance, manual review, promotion | Runtime/migration domains, exact archives, or exact attestations as declared |

A version-only metadata change therefore reruns packaging, exact ZIP loading, and upgrade evidence while allowing unchanged gameplay scenarios to reuse their existing domain-bound proof. A gameplay, settings, migration, dependency-contract, fixture, harness, or Factorio-binary change invalidates only the scenarios that declare that input, except unknown inputs, which invalidate all scenario domains. The actual 3.2.0 candidate is not version-only because its explicit target-capability guard changes the packaged data domain.

## Evidence And Gate

Evidence is stored under `artifacts/assurance/evidence/<safe-test-id>/<fingerprint>/`. Each directory may contain an expiring `running.json`, immutable attempt records, or atomic `passed.json` and `blocked.json` pointers. Successful execution requires a structured `mir-test-result-v1` whose assertion IDs, statuses, and artifact descriptors exactly match the capsule. The protected release workflow uses separate F0 through F4 jobs and a fresh no-reuse plan. It transfers only the exact candidate and plan authority to workers, collects one immutable capsule delta per worker, restores the shared ledger once at the gate, and admits only protected-release producer attestations to release sealing.

The aggregate gate does not trust job names, mutable status, or an arbitrary passed summary. It reconstructs the canonical plan, recomputes the candidate domain manifest, requires exact trusted evidence for every planned fingerprint, writes `artifacts/assurance/evidence-bundle.json`, and fails if any test is missing, extra, blocked, malformed, untrusted, stale for a forced run, or bound to different candidate bytes. Sealing invokes that canonical gate and binds the plan material, required set, candidate descriptor, evidence bundle, capsule set, exact runtime-performance evidence, exact manual package-review attestation, verifier release, and protected producer attestation.

## Backport Rule

The Factorio 2.0 `2.5.0` companion receives the portable verifier and CI model, but it recalculates every fingerprint using its own candidate ZIP, target profile, Factorio 2.0 binary, scenario declarations, fixtures, dependency contract, and `2.4.9` upgrade source. Evidence from Factorio 2.1 cannot satisfy a Factorio 2.0 gate.

## Acceptance

- `scripts/Test-MIRAssurance.ps1` passes.
- A fast plan emits stable fingerprints and explicit dispositions.
- An isolated exact-ZIP Factorio scenario passes through `-ScenarioWorker`.
- Rerunning an unchanged test produces `REUSE`.
- A version-only change preserves the gameplay dependency-contract hash while changing artifact and metadata hashes.
- The default GitHub workflow exposes `MIR / verification-gate`.
- Static validation passes, subject to separately classified user-owned distribution changes.
- The exact MIR 3.1.9 archive upgrades to the exact MIR 3.2.0 candidate with settings, native-owner technology level, current research, fractional progress, and fixture state retained.
