# 3.0.0 Transition Checkpoint 3

Updated: 2026-07-07

This checkpoint records the MIR 3 transition state before deeper compiler
movement out of legacy orchestration. It does not declare `3.0.0` release
readiness. It exists so later planner, index, capability, and emitter migration
can be compared with a named, validated midpoint.

## Completed Since Checkpoint 2

- `prototypes/mir/platform/factorio/data_raw.lua` is the platform adapter for
  `data.raw`, prototype lookup, technology lookup, and `data:extend`.
- `prototypes/mir/emit/technology_builder.lua` emits through the platform
  adapter instead of calling `data:extend` directly.
- `prototypes/mir/report/decision_export.lua` is the report-layer bridge for
  generated technology `DecisionRecord` rows.
- `prototypes/mir/compatibility/claim_registry.lua` mirrors the report-facing
  subset of compatibility claims and identifies `.mir/compatibility.yml` plus
  `fixtures/compat-matrix/claims.json` as authority sources.
- `prototypes/mir/compatibility/overlays/air_scrubbing.lua` is declarative
  Air Scrubbing policy data, and the legacy stream table consumes it.
- `prototypes/mir/settings/` owns settings registry, visibility evaluation,
  builder helpers, and the legacy adapter.
- Optional stream setting visibility uses `ui_visibility` metadata and active
  mods only. It does not inspect final prototypes during the settings stage.
- Hidden unavailable stream settings remain registered and do not use
  `forced_value` by default.
- `.mir/settings.yml` records stable setting ID, hidden-not-deleted, backport,
  and no-forced-value policy.
- `scripts/Test-MIRSettingsVisibility.ps1` lints the settings visibility
  contract.
- `mir-fixture-assert-hidden-setting-readability` proves hidden optional-stream
  startup settings remain readable during `data-final-fixes.lua`.
- `scripts/Test-MIRPolicyLints.ps1` now checks that claim levels are documented,
  `.mir/streams.yml` entries exist in the canonical manifest, `.mir` target IDs
  exist in the Lua claim registry, and fixture claim mods are mirrored in Lua.

## Current Architecture State

This is now a transitional MIR 3 compiler architecture, not only a docs shell.
The repo has real module boundaries, a platform adapter, an emitter adapter,
decision export, settings governance, claim registry, overlay policy data, and
fixture-backed visibility proof.
The current transition debt can be measured with
`.\scripts\mir.ps1 legacy inventory`, which writes JSON and Markdown reports to
`artifacts/legacy-inventory/`.

The main generation path is still transitional:

```text
root data-final-fixes.lua
  -> prototypes/mir/stage/data_final_fixes.lua
  -> prototypes/mir/stage/data_final_fixes_steps.lua
  -> MIR stream compiler / exact diagnostics registry / planner compiler
  -> legacy-compatible adapters where migration is still in progress
```

The intended destination remains:

```text
stage
  -> platform/index facts
  -> graph/classify/policy/capabilities
  -> planner
  -> DecisionRecord + StreamSpec
  -> emit
  -> report
```

## Known Remaining Gaps

- `prototypes/mir/stage/data_final_fixes_steps.lua` owns the transitional
  data-final-fixes step adapters, but several steps still call legacy-compatible
  prototype modules while their internals migrate.
- `prototypes/mir/planner/compiler.lua` owns the report compiler entrypoint,
  but still imports some old fact, capability, owner, and diagnostic modules.
- `prototypes/mir/index/registry_builder.lua` owns the fact registry
  implementation, but several helpers still bridge to older compatibility
  modules.
- The old `prototypes/compat/` profiles, owner detection, competing
  productivity cleanup, competing base-extension cleanup, family adoption, and
  compatibility planner paths are now one-line shims. Their implementations
  live under MIR compatibility, index, and policy modules.
- Air Scrubbing and ATAN Ash exact-recipe diagnostics now run through the MIR
  compatibility diagnostics registry, but broader compatibility targets are
  still claim/load-check records rather than executable overlays.
- `prototypes/mir/planner/stream_compiler.lua` owns the generated stream loop,
  but some stream construction responsibilities still need to move into
  capability-specific planners.
- `DecisionRecord` and `StreamSpec` exist but do not yet cover every generate,
  diagnose, reject, and observe row.
- The first legacy inventory baseline reports zero active `prototypes/compat`
  modules and zero generated streams missing manifest rows, while `prototypes/lib`
  and a small number of `prototypes/mir/legacy` modules remain active.

## Next Work

1. Route Air Scrubbing diagnostics through a MIR namespace while preserving row
   output.
2. Add a declarative ATAN Ash compatibility overlay and consume it from the
   stream definition.
3. Move exact overlay recipe-productivity emission through
   `StreamSpec -> technology_builder` in narrow slices.
4. Move fact registry construction behind `prototypes/mir/index/` shims.
5. Move planner compiler ownership into `prototypes/mir/planner/` and make old
   planner paths compatibility shims.
6. Keep regression row/hash comparison mandatory for any behavior-adjacent
   compiler migration.
