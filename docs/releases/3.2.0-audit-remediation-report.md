# MIR 3.2.0 Audit Remediation Report

## Verdict

This checkpoint materially hardens both the MIR compiler and its release verifier, including the Space Exploration startup-crash class that triggered the latest compatibility report. It remains a development checkpoint on `dev`, not a release-qualified 3.2.0.

The canonical published baselines remain MIR 3.1.9 for Factorio 2.1 and MIR 2.4.5 for Factorio 2.0. MIR 2.5.0 does not yet exist as a release or final candidate. Its backport starts only after the canonical 3.2.0 source freeze.

No tag, release seal, promotion, public upload, or 2.5.0 backport was created in this remediation.

## Scope And Release Authority

The release records now use `.mir/releases.json` as the canonical authority. It identifies:

- `3.1.9` on `main` as the published Factorio 2.1 baseline;
- `2.4.5` on `legacy` as the published Factorio 2.0 baseline;
- `3.2.0` on `dev` as an unreleased development candidate;
- `2.5.0` on `tmp/2.0` as a planned backport after the 3.2 source freeze, with no archive.

The branch policy, release wave, promotion default, distribution inventory, release notes, and maintainer queue now agree with that authority. The invented or temporary `1.9.5` and `2.5.0` distribution rows were removed. The tracked root distribution inventory contains 44 archives and includes the current 3.2.0 development archive without treating it as a tagged release.

`Test-MIRReleaseAuthority.ps1` makes disagreements among those views a static validation failure.

## Compiler And Prototype Integrity Changes

### Central effect contracts

`prototypes/mir/integrity/effect_contracts.lua` is now the shared authority for effect identity and target validation. Registered target-aware contracts cover:

- `change-recipe-productivity`;
- `unlock-recipe`;
- `unlock-space-location`;
- `give-item`;
- `ammo-damage`;
- `gun-speed`.

Generation planning, relationship indexing, sanitation, and postconditions consume this authority instead of carrying independent identity rules.

External target-bearing effects with impossible targets are removed deterministically and diagnosed. MIR-proposed effects with impossible targets reject their operation. Unknown effect types are retained and diagnosed rather than guessed at.

This directly covers the Space Exploration report in which `kr-copper-cable-from-copper-ore` was removed after another technology still referenced it. The regression fixture contains a valid and dangling unlock effect and proves that only the impossible effect is removed.

### Whole-graph safety

`prototypes/mir/planner/technology_graph.lua` builds the combined existing and planned technology graph and uses iterative Kosaraju analysis in `O(V + E)`. It records bounded witnesses and classifies:

- MIR-only cycles;
- mixed MIR/external cycles;
- external-only cycles;
- missing prerequisites;
- disabled prerequisites;
- unavailable research mechanisms;
- unavailable labs or science ingredients.

Unsafe MIR operations are converted into typed skip decisions and propagated to dependent planned operations. Duplicate identities and duplicate generated names remain fatal compiler-contract violations because they indicate internal corruption rather than an expected compatibility rejection.

### Explicit construction and mutation ownership

Base continuations no longer begin with `deepcopy(base_tech)`. The new base-extension builder copies only supported, intentional fields.

`prototypes/mir/planner/stream_compiler.lua` now plans stream operations. `prototypes/mir/emit/stream_executor.lua` owns their mutation. Generation-plan publication is delayed until authoritative postconditions complete.

### Index and traversal work

Technology researchability is iterative. Recipe-output discovery uses indexed read-only views instead of repeated recipe scans and whole-fact deep copies on each lookup. Compiler telemetry records recipes, technologies, effects, graph edges/components, cyclic components, recipe scans/copies, candidate/accepted/rejected operations, diagnostics, and snapshot/graph/planning/postcondition phases.

## Verification Ledger Changes

The verification ledger now uses:

- test, plan, result, capsule, bundle, and seal JSON schemas;
- evidence schema 4 and bundle schema 2;
- exact in-process hashes for tracked, modified, and untracked inputs;
- immutable attempt capsules with atomic `passed.json` or `blocked.json` pointers;
- quarantine for corrupt pointers;
- structured result and artifact validation;
- exact candidate bytes and content descriptors;
- exact external mod closure and Factorio installation descriptors;
- canonical plan reconstruction at every worker and aggregate gate;
- per-test freshness bindings;
- protected release-producer validation;
- a true F0-F4 workflow dependency chain.

The verifier’s own schema directory is part of the trusted runner fingerprint. The release profile includes the static contracts that were previously run outside the aggregate.

## Target And Governance Changes

The Factorio 2.0 profile now treats mod-data as unsupported and disables productivity-family adoption that depends on it. Target validation rejects enabled features whose declared capability dependencies are unavailable.

`.mir/modules.yml`, `.mir/compatibility.yml`, `.mir/fixtures.yml`, `.mir/performance.yml`, `.mir/performance-budgets.json`, `.mir/branches.yml`, `.mir/release-wave.yml`, `.mir/assurance.json`, `.mir/targets.json`, and `.mir/distributions.json` were updated to match the code and release posture.

The package identity scanner and assurance input resolver ignore `.mir/target-lines` as historical snapshots rather than active implementation. A regression proves that changing a historical snapshot does not alter the active package/harness identity.

## Issues Found During Remediation

### Synthetic recipe collision exposed over-broad fixture sanitation

An initial test implementation sanitized a synthetic shared recipe too early and changed the collision fixture’s intended ownership conditions. The fixture and sanitation boundary were corrected so it tests target integrity without erasing the collision contract.

### Publication command classification rejected a valid postcondition

The generation-integrity fixture initially rejected the new delayed publication command because its allowed command-kind set still reflected pre-refactor ordering. The assertion was updated to recognize publication as a governed postcondition command.

### Rejection propagation selected the wrong reason

Dependent planned operations could inherit a research-ingredient rejection before the more fundamental missing-prerequisite rejection. The propagation priority was corrected so structural prerequisite failures remain the primary typed reason.

### Edit planning escalated because a new manifest was unclassified

The first edit plan treated `.mir/compiler-contract-coverage.yml` as unknown and expanded into `static.full` plus the runtime matrix, exceeding the intended fast-check window. It is now explicitly classified as compiler/data-stage policy. The regenerated edit plan contains only `static.architecture` and `static.compiler`.

### Local fresh evidence used process IDs as run IDs

The focused workers passed independently, but the gate rejected them because plan, worker, and gate ran in separate PowerShell processes and each process generated a different `local-$PID` identity. Fresh local evidence now adopts the validated plan-owned run ID and attempt. Protected CI workers cannot override identity and must match the actual protected workflow run.

### Producer cloning serialized a PowerShell dictionary wrapper

After the run-ID correction, worker capsules carried the right run ID but still failed trust validation. The generic object-clone helper serialized an ordered dictionary as `Count`, `Keys`, `Values`, and `SyncRoot` properties, nesting the real verifier and policy attestation. Dictionary cloning now copies actual dictionary keys, and the assurance self-test rejects wrapper-shaped producer records.

### The allowlisted builder retained a forbidden legacy field

The first repository-wide static gate found `icon_mipmaps` in the new base-extension presentation allowlist. That field is not portable across MIR target lines. It was removed, and the existing repository-wide prototype guard prevents it from returning.

### A static sentinel still required the old sanitizer API

The next static pass found that the validation facade still required `sanitize_registered_technology_effects()` even though the new compiler runs generic `sanitize_all_technology_effects()` through the pipeline and central target contracts. The sentinel now verifies the new pipeline call and target lookup authority.

### Changelog wording violated Factorio changelog policy

The static gate rejected wrapped bullet continuations and internal validation/proof wording. The 3.2.0 changelog entry was shortened to valid, player-facing single-line bullets under the 132-character limit.

## Backport Boundary

The 2.5.0 backport must be cut from the final frozen 3.2 source, not from this intermediate checkpoint. It must preserve canonical Factorio 2.0 capabilities, apply only target-specific cuts and adapters, use an exact 2.0 binary and 2.4.5 upgrade source, and produce independent evidence and a separate seal.

Until that point, 2.5.0 remains a planned version with no authoritative archive.

## Development Recommendation

Continue 3.2 on `dev` in this order:

1. behavior-equivalence export and approved deltas;
2. explicit snapshot/compiler context and remaining cache migration;
3. complete planner/emitter separation;
4. 100k-scale and deterministic-order campaigns;
5. exact ecosystem and upgrade campaigns;
6. final F0-F4 qualification and protected sealing;
7. source freeze, then the 2.5.0 backport.
