# MIR 3.1.0 Migration Guide

## Upgrading From 3.0.5

MIR 3.1.0 targets the same Factorio `2.1` line as 3.0.5. Replace the old archive through Factorio's normal mod update flow or place the new archive in the mods directory, then load the existing save.

No settings reset or research reset is required. The release preserves generated technology IDs, released setting IDs and defaults, profile schema, runtime namespaces, and existing migrations. The exact automated upgrade proof retained a non-default scripted stream checkbox, a non-default effect value, completed infinite-research levels, fixture storage, and the applied runtime multiplier.

## Visible Behavior

The default player contract remains compatible with 3.0.5, but 3.1 replaces the generation architecture and adds a conservative automatic compiler. Players should expect:

- deterministic recipe/science planning and prerequisite rejection;
- recycler-loop safety for probabilistic, byproduct, fluid, variant, and conversion paths;
- scripted effects across save/load, force changes, setting disable, and re-enable;
- structural manufacturing discovery for modules, loaders, belts, drills, inserters, furnaces, assemblers, labs, solar panels, and storage entities;
- safe attachment to existing stable technologies by default, with reviewed generic assembler/lab generation available only through the opt-in `safe-generate` mode;
- complete diagnostic accounting for recipes the compiler covers, adopts, skips, rejects, or cannot classify;
- deterministic release archives and clearer compatibility diagnostics.

MIR continues to skip a stream when its required science or prerequisite chain cannot be researched by the active labs. A conservative skip is preferable to generating an unavailable technology.

## Settings Profiles

Existing profile strings remain valid. Export a profile before changing a large modpack if you want a portable backup, but importing it again is not part of the normal 3.0.5-to-3.1.0 upgrade.

The settings page retains released IDs even when a target/provider-specific row is hidden. Removing or adding a provider mod should not silently discard the stored value.

The new `mir-automatic-compiler-mode` setting defaults to `safe-attach`. `off` disables family handling, `report` records decisions without attachment or generation, `safe-attach` may attach structurally proven recipes to existing stable streams, `safe-generate` additionally enables the two reviewed generic assembler/lab streams, and `exact-pack` is reserved for reviewed compatibility-pack behavior. Missing safety proof always blocks generation regardless of mode.

## Modpack Notes

AAI, BZ Space Age, Bob 3.0, Krastorio 2, K2SO, and a 46-mod Space Age planet cluster have exact-archive local load observations at their documented claim level. K2SO's overhaul-specific lab graph causes MIR to skip several direct-effect streams; this is intentional conservative behavior and is not a broad K2SO support claim. Angel, Space Exploration, and Pyanodon remain unclaimed until their complete dependency closures are executed.

If an existing save reports an unavailable generated technology after upgrade, keep the save and mod list unchanged, export the MIR profile, and attach `factorio-current.log` plus the generated compatibility report to the issue. Do not remove technologies or edit the save as a first response.

## Backport Lines

MIR 3.1.0 is not compatible with Factorio 2.0. Factorio 2.0 remains on the `2.x` release line; the companion release derived from accepted 3.1 contracts is planned as 2.4.0.
