# 3.2.0 Technology Design Foundations Report

## Outcome

This `dev` change set closes the four concrete correctness defects identified after checkpoint `d0bae36`, persists content-addressed compiler evidence, and implements the first compatibility-preserving `TechnologyDesign` vertical slice. It is a development checkpoint, not a release qualification or ecosystem-wide support claim.

No released technology name, stream manifest identity, effect, science selection, prerequisite, cost, icon, migration policy, or generated-registry identity was intentionally changed. The stable golden plan still contains 70 baseline and two predeclared 3.1 automatic-family identities.

The preceding broad startup-crash and verifier work remains documented in `docs/releases/3.2.0-audit-remediation-report.md`.

## Correctness repairs

### Pre-index and post-emission sanitation

The pipeline now runs `sanitize-input-technology-effects` immediately after exact compatibility repairs and before module permissions, prototype limits, indexes, snapshots, or planning. This prevents dangling external unlocks and other missing effect targets from influencing recipe unlocks, science reachability, prerequisite selection, family classification, and coverage.

The final sanitation/assertion pass remains in place, but now runs after all emission and mutation commands and before compatibility reports, output parity, and publication. It catches MIR output defects and unexpected late mutations without allowing reports to describe unsanitized state.

Both passes produce schema-1 ledgers. Each affected technology row records:

- original technology name and effect count;
- original indexes for removed effects;
- effect type, target, and rejection reason;
- generated or external owner kind;
- owning mod when MIR can know it, otherwise explicit unknown status;
- exact retained original effect order.

### Actual SCC cycle evidence

Technology SCC diagnostics no longer join sorted component members into a possibly false path. A deterministic iterative DFS now finds prerequisite edges inside the SCC. Each cyclic component publishes:

- `component_id`, fingerprinted from sorted members;
- bounded `member_sample`;
- bounded `actual_cycle_witness`;
- an explicit witness-truncation flag;
- compatibility aliases for the previous node and witness fields.

The compiler-contract fixture proves the non-lexical cycle `A -> C -> B -> A` and verifies every adjacent witness edge exists.

### Semantic rejection priority and contributing causes

Rejections now use the explicit semantic priority table requested in the assessment. Missing and disabled prerequisites outrank MIR, mixed, and external cycles; those outrank effect-target and research-mechanism failures; lab and unreachable-science failures follow.

Every rejected planned technology retains:

- a backward-compatible top-level `code`, `reason`, and evidence list;
- a structured `primary` reason;
- every distinct subordinate `contributing` reason.

Propagation deduplicates reasons by code, origin, and evidence. SCC reasons share the component fingerprint as their origin, preventing quadratic duplication inside large components.

### Semantic versus operational fingerprints

`CompilationPlan.semantic_fingerprint` now covers only semantic compiler material: source snapshots, the input sanitation ledger, finalized operations, stream and base plans, graph decisions, rejection reasons, and expected output. The existing `fingerprint` field remains an alias for compatibility.

Telemetry is attached separately as `telemetry_fingerprint` and a schema-1 run record. Timing, counters, and witness-order changes can alter run evidence but cannot alter semantic plan identity. The compiler-contract fixture finalizes equivalent plans against changing telemetry and proves semantic identity remains stable.

## Explicit per-run compiler context

Pipeline dependency state and sanitation artifacts now live in an explicit schema-1 compiler context created for each command-DAG execution. The old module-level completed-command table was removed. Context artifacts are once-only records and are deep-copied on write, read, and snapshot.

This begins, but does not complete, the assessment's universal immutable context work. Planner, index, telemetry, and registry phase singletons still require an epoch/removal audit.

## Persistent compiler evidence

Postcondition publication now emits `more-infinite-research-compiler-evidence` as mod-data. The schema-1 artifact contains:

- semantic compilation fingerprint;
- independent telemetry fingerprint;
- input and output sanitation ledgers;
- a separate fingerprint for each ledger;
- a fingerprint for the complete evidence envelope.

The generation-integrity fixture verifies that the published semantic fingerprint matches the current `CompilationPlan` and that both ledger passes and all content fingerprints exist.

Later pre-release hardening retained that complete evidence as the separately named internal diagnostic artifact and changed the normal-load public prototype to a compact schema-1 projection. The current publication contract is governed by `docs/reference/schemas/compiler-evidence.md`; this section records the earlier foundations checkpoint rather than the final public payload shape.

## TechnologyDesign schema-1 vertical slice

Fixed and automatic stream candidates now enter one normalized representation after cross-stream effect ownership arbitration. The record separates:

- evolving `candidate_id` from released `technology_id`;
- capability, family, and partition identity;
- recipe, item, fluid, and entity membership;
- identity, effects, progression, cost, presentation, ownership, and runtime contracts;
- proof gates and compiler context;
- source, provider, family, and leaf-field provenance;
- independent safety, discovery, design, validation, applicability, identity, runtime-action, and public-claim maturity axes.

Every major design dimension has a provenance/lock envelope. `provenance.fields` additionally records independent leaf paths such as technology ID, effects, science, prerequisites, cost formula, maximum level, localized name, icon fallback, ordering, migration policy, and upgrade behavior. Stable fixed IDs can therefore remain locked while science, prerequisites, settings-derived cost, and icon fallbacks remain adaptive.

`CompilationPlan` derives expected stream prototype shape from `TechnologyDesign`, and the stream emitter consumes the same record. This is an active vertical slice, not metadata attached beside the old emitter path.

Automatic family identities now carry the resolved family IDs and provider IDs available from the family resolver. Existing automatic technology IDs remain released because they were already present in the 3.1 stable manifest; this refactor does not silently rename or demote them.

## Known limits and next development work

Highest-priority continuation on `dev`:

1. Complete Phase A context work by inventorying and removing or epoching remaining planner, index, registry, diagnostic, and telemetry singletons.
2. Finish planner/presentation/emitter separation without changing normalized output.
3. Establish the Phase B 3.1.9 normalized-design baseline and governed `approved-delta/3.1.9-to-3.2.0.json` before expanding structural refactors.
4. Complete Phase C by adapting base continuations and native-owner adoption to `TechnologyDesign`, then require the IR for every materializing plan row.
5. Add candidate qualification, approval, quarantine, promotion, demotion, and migration records without coupling candidate IDs to released technology IDs.
6. Build review catalog/diff tooling before rule induction or broad automatic generation.
7. Run focused automatic-family, base-extension, adoption, large-graph, and randomized-order campaigns before broader ecosystem qualification.
8. Defer the 2.5.0 backport until the 3.2 behavior baseline, approved delta, and source lock are stable.

External mod ownership remains unknowable for ordinary Factorio prototype tables unless a producer supplies provenance; the sanitation ledger records `<unknown>` and `owning_mod_known = false` instead of inferring from names. Cycle witnesses longer than the governed bound retain only a real-edge prefix and set the truncation flag rather than inventing a closing edge.
