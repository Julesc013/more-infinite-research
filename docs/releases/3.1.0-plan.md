# MIR 3.1.0 Plan

MIR 3.1.0 is a bounded correctness, architecture, maintainability, performance, and capability-expansion release. It preserves the 3.0.5 public contract unless a tested migration explicitly changes it.

## Entry Gate

- 3.0.5 is published and frozen.
- 2.3.5 exact bytes are qualified, integrated into `legacy`, and tagged. External upload remains separately observable.
- Accepted portable lessons are recorded in `.mir/portable-return.yml`.
- `pre-3.1-modernization` identifies the exact implementation baseline.

On 2026-07-12 the maintainer authorized continuous execution through gates without pausing. This disposition permits package-visible 3.1 implementation after the exact 2.3.5 archive was qualified, integrated, and tagged; it does not represent the unperformed normal-UI observation or external upload as completed evidence.

## Protected Contracts

- Generated technology IDs and migration meaning.
- Released startup setting IDs, enum values, and defaults.
- Settings profile compatibility.
- Persisted runtime namespaces.
- Existing save progression.
- Default 3.0.5 player behavior.

## Explicit Dispositions

- Recycling-safe cloned variants: bounded protocol prototype only if stable IDs, exact cloning, unlock/effect parity, migration, UI, and performance evidence all exist; otherwise defer.
- Dedicated safe machines: defer until safe variants are proven.
- Dynamic module-effect research: bounded native/event-driven investigation; reject prototype-per-level, invisible-beacon, per-tick scan, and hidden-proxy designs.
- Dynamic productivity-cap research: reject from core unless Factorio exposes a clean native model.
- Machine crafting-speed research: investigate native or bounded force-level APIs only.
- Broad overhaul campaigns: defer to 3.3+ fixture campaigns.
- Full graph/SCC intelligence: 3.2, not 3.1.

## Implementation Order

1. Characterize 3.0.5 defaults, IDs, profiles, runtime state, and reports.
2. Build immutable facts and indexes.
3. Introduce typed descriptors and positive target requirements.
4. Separate policy decisions from mutation commands.
5. Add missing boundary fixtures and declarative scenario inputs.
6. Decompose validation and make package construction deterministic.
7. Establish performance budgets.
8. Complete the narrow capability lifecycle slice.
9. Run upgrade, exact-dist, local-library, freshness, and manual gates.

## Non-Goals

No ground-up rewrite, generic public plugin host, universal overhaul claim, full graph-intelligence engine, or unbounded feature intake belongs in 3.1.
