# MIR 3.1.0 Automatic Compiler Development Summary

The first automatic family compiler implementation is complete on `dev` from the published immutable 3.0.5 behavior baseline. It is part of unreleased MIR 3.1.0. Earlier 3.1.0 candidate artifacts are superseded and must be rebuilt and requalified from the final refactor source before publication.

## Implemented

- All fixed streams compile into one pure `GenerationPlan`, validate as a complete set, then emit.
- Existing productivity-family adoption is also planned before mutation.
- RecipeFactV2 preserves typed variants, probability, catalysts, productivity exclusions, surfaces, caps, and source class.
- Shared relationship indexes cover recipe I/O/category/unlocks, item placement, entity types, effect ownership, labs, modules, upgrades, subgroups, and surfaces.
- Data-only FamilyRules attach structurally proven loader/logistics, mining-drill, furnace, inserter, power, and module recipes to six existing stable streams.
- Assembling-machine and lab manufacturing use predeclared stable generic-family identities and remain gated behind `safe-generate`; the default `safe-attach` mode emits neither technology.
- The golden plans lock the 70 existing candidate IDs plus the two reviewed 3.1.0 family IDs.
- Fixture-only competing policy moved out of production profiles and into validated `CompatibilityPack` `mod-data` packets.
- Capability resolver schema 2 uses `materialize` and `result`; resolver reporting is no longer described as prototype emission.
- Pipeline commands declare phases and dependencies, reject cycles, and fail on out-of-order execution.
- Direct-effect identity and numeric metadata have one domain authority.
- Runtime scenario schema 2 stores full target, kind, group, surface, and feature records. Compatibility campaign schema 2 separately owns target, setup, roots, settings, expected plan boundary, timeout, and claim level; the audit runner resolves line-specific manifests automatically.
