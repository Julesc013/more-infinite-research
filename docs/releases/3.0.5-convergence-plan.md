# MIR 3.0.5 Convergence Plan

## Objective

MIR 3.0.5 is the definitive compatibility-hardening patch for the 3.0 line. It
preserves the modern Factorio 2.1 compiler and imports accepted behavior from
the backport rings without importing their implementations wholesale.

```text
3.0.5 = dev baseline
      + accepted portable behavior
      + adapters, invariants, fixtures, and provenance
      - duplicate target knowledge
      - accidental historical behavior
      - lower-target metadata and feature cuts
```

The frozen baseline is `pre-3.0.5-synthesis` at commit `b9b1abc`. Accepted and
rejected behaviors are recorded in `.mir/convergence.yml` and the
[backport behavior ledger](../compatibility/backport-ledger.md).

## Ordered Slices

1. Record immutable 1.7.0 release evidence.
2. Port and strengthen science prerequisite reachability.
3. Generate Lua and PowerShell target views from one canonical profile.
4. Make runtime state backend selection explicit.
5. Add validation scenario groups and persistent structured summaries.
6. Separate settings codec coverage from reduced settings visibility.
7. Make conditional weapon overlap depend on exact replacement coverage.
8. Reconcile TODO, public copy, architecture docs, and changelog.
9. Centralize recipe unlock facts and reject science self-locks.
10. Restrict weapon cleanup to MIR's registry and prove external science.
11. Bind candidate evidence to source, profile, groups, package, and summary.
12. Run complete Factorio 2.1 base, Space Age, and exact-dist gates.

Each slice is independently reviewable. No slice may change `factorio_version`,
the Factorio 2.1 dependency floor, generated technology IDs, modern feature
availability, or published archive bytes.

## Architecture Boundaries

- `.mir` owns target and convergence policy.
- `platform/factorio` owns target selection and runtime backend access.
- `capabilities/science_integration` owns science recipe availability and
  unlock inference.
- planners consume validated prerequisites but do not invent target policy.
- `emit` remains the only generated technology mutation boundary.
- validation modules own orchestration metadata and evidence aggregation.
- compatibility policy may classify external weapon owners but does not create
  generated technologies.

## Release Gates

- static, governance, architecture, locale, package, and manifest validation;
- complete Factorio 2.1 base and Space Age runtime profiles;
- enabled, existing, acyclic, reachable generated prerequisites;
- deterministic science unlocker selection;
- direct science self-lock and pack-production cycle rejection;
- exact replacement coverage for conditional weapon cleanup;
- no mutation of similarly named external weapon continuations;
- separate settings codec and reduced-surface fixtures;
- exact-dist base and Space Age package smoke;
- a final structured summary with no failed, running, or incomplete required
  scenarios;
- candidate freshness against exact source, profile, required groups, package
  content, and structured-summary identity;
- no lower-target metadata, defaults, assets, or feature cuts on `dev`.

## Deferred Work

Factorio 0.16 science identifiers and effect proof belong to `tmp/0.16` after
3.0.5. New overhaul support and any public extension API belong to 3.1 or a
later correctly versioned release. Archive and museum lines do not block 3.0.5.

## Release Rule

Build the candidate from a clean checkout, validate the exact archive, and tag
the commit whose bytes were tested. Existing saved startup settings retain
their stored values. Published historical archives remain immutable.
